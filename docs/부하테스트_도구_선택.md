# 부하테스트 도구 선택: k6

## k6란?

Go 기반의 오픈소스 부하테스트 도구. JavaScript로 테스트 시나리오를 작성하고 CLI에서 실행한다.

### 설치

```bash
brew install k6
```

### 테스트 스크립트 예시

```javascript
import http from 'k6/http';
import { check } from 'k6';

export const options = {
  vus: 100,          // 동시 가상 유저 수
  duration: '10s',   // 테스트 시간
};

export default function () {
  const res = http.post('http://localhost:8080/coupon/1/issue?userId=' + __VU);
  check(res, {
    '200 OK': (r) => r.status === 200,
  });
}
```

### 실행

```bash
k6 run test.js
```

### 결과 예시

```
✓ 200 OK

http_req_duration ... avg=12.5ms  min=1.2ms  max=230ms  p(95)=45ms
http_reqs .......... 15000
iterations ......... 15000
vus ................ 100
```

주요 지표: TPS(http_reqs/duration), 응답시간(avg, p95), 에러율로 Phase별 before/after를 비교한다.

## 동시성 모델 비교


| | k6 | nGrinder | JMeter |
|---|---|---|---|
| 동시성 모델 | Go 고루틴 | OS 스레드 (Java) | OS 스레드 (Java) |
| VU당 메모리 | ~수 KB | ~1MB | ~1MB |
| 로컬 한 대 한계 | 수만 VU | 수천 VU | 수천 VU |
| 스케일링 방식 | 머신 한 대로 충분 | 에이전트 분산 (스케일 아웃) | 머신 추가 (스케일 아웃) |

- **JMeter, nGrinder**: VU 1개 = OS 스레드 1개. 스레드가 무거우니까 머신을 늘려서 해결한다.
- **k6**: VU 1개 = Go 고루틴. 고루틴이 가벼우니까 머신 한 대로도 충분하다.

JMeter, nGrinder 모두 Java 스레드 기반으로 설계됐다. 가상 스레드 대신 무거운 OS 스레드를 사용한다.

### 출처
- **JMeter**: [User's Manual - Test Plan](https://jmeter.apache.org/usermanual/test_plan.html) — "Each thread corresponds to a Java thread, further utilizing an Operating System (OS) thread for its execution."
- **nGrinder**: [Wiki - Virtual User](https://github.com/naver/ngrinder/wiki/Virtual-User) — "Virtual Users in nGrinder is mapped into threads."
- **nGrinder**: [Wiki - Process and Thread](https://github.com/naver/ngrinder/wiki/Process-and-Thread) — "Each thread acts like a user."

## 비교 대상

| | k6 | nGrinder | JMeter |
|---|---|---|---|
| 스크립트 언어 | JavaScript | Groovy/Jython | XML (GUI) |
| 엔진 | Go | Java (JVM) | Java (JVM) |
| 리소스 소비 | 가벼움 | 무거움 (서버 필요) | 무거움 |
| Git 추적 | JS → diff 깔끔 | 코드 → 괜찮음 | XML → 힘듦 |
| 설치 | `brew install k6` | 별도 서버 설치 | JAR 실행 |
| 분산 테스트 | 클라우드 (유료) | 내장 (컨트롤러+에이전트) | 직접 구성 |
| 결과 리포트 | CLI 텍스트 | 웹 UI | GUI |

## 선택 이유

### 1. 가볍다
Go 기반이라 JVM 도구들보다 리소스 소비가 적다. 로컬에서 반복 실행하기 좋다.

### 2. Git 관리가 편하다
테스트 스크립트가 JS 파일이라 git diff로 변경 이력 추적이 쉽다.

### 3. CLI 기반
GUI 없이 터미널에서 바로 실행하고 결과를 확인할 수 있다.
동일 스크립트를 Phase별로 반복 실행해서 수치 비교하기 편하다.

### 4. 설치가 간단하다
`brew install k6` 한 줄이면 끝난다. nGrinder처럼 별도 서버를 띄울 필요 없다.

## 탈락 이유

- **JMeter**: XML 기반 GUI 도구라 git 추적 힘들고, JVM이라 무거움
- **nGrinder**: 컨트롤러 서버를 별도로 설치해야 해서 혼자 로컬에서 공부하기엔 과함
